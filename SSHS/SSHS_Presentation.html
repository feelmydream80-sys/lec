Problem Statement
왜 급식 영양 분석이 필요한가?
학교 급식 정보의 한계와 학생 건강 관리의 현실
📋
단순 칼로리 정보만 제공
NEIS 급식 데이터는 "총 칼로리"만 알려줄 뿐, 단백질·지방·칼슘·철분·비타민C 등 6대 영양소의 균형은 파악할 수 없습니다.
⚠️
알레르기 식별의 어려움
메뉴 속 알레르기 유발 식품(18종)을 매일 수기로 확인해야 하는 불편함. 실수 시 응급 상황으로 이어질 수 있습니다.
📝
수기 기록의 비효율성
학생이 먹은 급식을 일일이 메모해 영양사나 부모와 공유해야 하는 번거로움이 있습니다.
📉
영양 불균형 사전 예방 불가
일주일/한 달 단위로 어떤 영양소가 부족하고 과다한지 객관적 데이터 없이 주관적 판단에 의존합니다.

Core Feature 01
AI 식판 영역 자동 분할
YOLO-seg로 6개 식품 구역을 실시간 분할 · 번호 매핑
1

영상 전처리 (OpenCV)

fastNlMeansDenoisingColored로 노이즈 제거 → Sharpening Kernel(3×3 라플라시안)으로 경계 강화

2

YOLO-seg 추론 (Ultralytics)

모델 best.pt로 conf_threshold=0.15, imgsz=500 설정 추론
클래스 0(식판) + 클래스 1~5(5개 음식 구역) 마스크 동시 검출

3

구역 번호 자동 할당

식판 바운딩 박스 내 Y좌표 기준 상/하 행 분리 → 상행 좌→우 1,2,3,4번 · 하행 좌→우 5,6번
Centroid(moment) 계산으로 중심점 정확히 추정

4

오버레이 시각화

6색 구역별 오버레이(투명도 25%) + 외곽선(두께 3px) + 섹션 번호 렌더링
JPEG 품질 85%로 Base64 인코딩 → 프론트 실시간 미리보기

07
Core Feature 02
픽셀 → 부피 → 무게 → 칼로리
OpenCV 투시변환 + 실제 식판 크기 기반 정밀 영양소 계산 알고리즘

🔢 계산 파이프라인

  • 픽셀 면적 → YOLO 마스크 픽셀 수 측정
  • 실제 크기 → Perspective Transform(35cm×25cm) 보정
  • 실제 면적 = 픽셀 면적 × (cm²/px²) 비율
  • 부피 = 면적 × 구역별 높이(1.0~2.0cm, config 기반)
  • 무게 = 부피 × 밀도(0.8g/cm³, 기본값)
  • 칼로리 = 무제 × (kcal/100g) ÷ 100

📐 투시변환 (Perspective Transform)

  • 사용자가 4점 클릭 또는 자동 모서리 감지
  • cv2.getPerspectiveTransform + cv2.warpPerspective
  • 출력 크기 600×450px 고정 (종횡비 보존)
  • 비틀어진 사진 → 정면 평면 보정

🔗 영양소 API 매칭

  • 구역별 음식명 → 공공데이터포털 API 호출
  • "음식" 분류 + "외식" 제외 필터링
  • 100g당 kcal 및 6대 영양소 반환
핵심 파라미터
식판 실제 크기: 35cm × 25cm │ 구역별 높이: 1.0~2.0cm │ 기본 밀도: 0.8g/cm³ │ 영양소 항목: 칼로리·탄수화물·단백질·지방·칼슘·철분·비타민C

AI Prompt Design
프롬프트 설계 — 어떻게 & 왜?
Gemini 2.0 Flash API에 전달하는 프롬프트 구조와 설계 의사결정
01
역할 부여 (Role Prompting)
"당신은 대한민국 학교 영양사이자 식품영양학 박사입니다" — 전문성 기반의 신뢰도 높은 조언 유도
02
맥락 데이터 구조화 (Context Injection)
개인화 정보(키/몸무게/BMI/성별/건강상태) + 6대 영양소 섭취율 + 알레르기 노출 + 이미지 분석 식사량을 JSON 형태로 주입
03
5가지 조언 템플릿 (Output Formatting)
